请精读这篇论文（PDF 链接）：【填入pdf的链接】

请生成一个内容详实、结构清晰、美观简洁、适合论文精读的 HTML 总结页面。最终交付一个可离线打开的 `.html` 文件。

---

## 0. PDF 获取与解析前置要求

在生成 HTML 前，必须先完成 PDF 获取与解析检查：

1. 优先访问我提供的 PDF 链接；
2. 如果 PDF 链接访问失败，尝试访问对应 arXiv / web page 页面并从中进入 PDF；
3. 如果网页解析失败，尝试直接下载 PDF；
4. 如果无法读取 PDF 正文或无法抽取图表，不要生成缺失图表的最终 HTML，而是明确说明失败原因。

生成 HTML 前必须确认：

* 已成功获取 PDF；
* 已能读取论文标题、作者、摘要、章节结构；
* 已能定位论文中的 Figure / Table / Algorithm / Equation；
* 已能从 PDF 页面中裁剪若干关键 Figure / Table；
* 不是只基于 arXiv 摘要或网页摘要生成。

图表抽取要求：

* 优先裁剪论文中的 Figure / Table 局部区域；
* 裁剪时保留标题、坐标轴、legend、表头和关键文字；
* 不要直接放整页 PDF 截图，除非只能 fallback；
* 如果无法精确裁剪，也要先给出局部区域裁剪图；
* 只有当 PDF 本身没有图表或图表不可见时，才允许说明“未能抽取”。

图像质量硬性要求：

* 严禁为了“看起来有图”而伪造论文原图、重绘复杂架构图、重绘实验曲线或重绘坐标图；
* 如果不能从 PDF/HTML 中获得清晰原图，就不要生成看似论文原图的替代图；
* 允许制作“概念解释图”，但必须明显标注为“概念示意图 / 非论文原图”，并且只能表达正文已经确认的数据流或模块关系；
* 概念图只能使用 HTML/CSS 或简单 SVG 盒子图，线条必须横平竖直或清晰折线，不允许交叉混乱、箭头指向错误、文字重叠或乱码；
* 坐标曲线、热力图、实验柱状图、定量结果图必须来自论文原图裁剪或 HTML 表格复刻，不能凭空重画；
* 所有图片必须在最终 HTML 中直接带可显示的 `src`，禁止使用 `data-svg`、空 `src`、运行时 JS 转换 base64 或懒加载占位图；
* 禁止把运行时生成的 SVG 当成论文图表。截图、导出、脚本失败或 CSP 限制时会导致空图、乱码或线条错位；
* 所有图片在插入前必须检查：文字可读、比例不变形、无乱码、无错位、无被截断、无过度压缩、移动端不溢出。

## 1. 总体目标

不要把 HTML 写成 checklist 式报告，而要写成一篇读者友好的论文精读文章。

页面应该带读者按以下理解路径阅读：

1. 这篇论文解决了什么问题？
2. 为什么这个问题重要？
3. 现有方法的核心瓶颈是什么？
4. 作者的核心 idea 是什么？
5. 方法 pipeline 如何从输入走到输出？
6. 每个关键模块为什么这样设计？
7. 训练目标如何支撑这些设计？
8. 实验如何证明方法有效？
9. 消融分别验证了哪些设计选择？
10. 方法有哪些局限？对后续研究或工程实践有什么启发？

内容组织必须围绕“理解路径”，不要机械复述论文章节。

---

## 2. 推荐页面结构

可以根据论文内容微调，但整体应遵循如下结构。

### 1. 论文概览

包括：

* 论文标题、作者、机构；
* 一句话总结论文贡献；
* 研究问题；
* 核心方法；
* 主要结果；
* 适合什么读者阅读；
* PDF / arXiv / Project / GitHub / OpenReview / Conference 等资源按钮。

首页第一屏重点是：

> 这篇论文为什么值得读，以及它的核心贡献是什么。

不要让大量元信息或未找到的链接喧宾夺主。

---

### 2. 投稿与资源信息

展示公开可查的元信息：

* arXiv 首次提交日期；
* arXiv 最新版本日期；
* PDF 链接；
* arXiv / web page 链接；
* Project Page，如有；
* GitHub / Code，如有；
* OpenReview / Conference / Workshop 页面，如有；
* 接收状态，如 Oral / Poster / Spotlight / Accepted，如公开可查；
* Submission Number，如公开可查。

如果查不到公开接收信息，明确写：

> 未找到公开接收信息。

不要编造 Project、GitHub、OpenReview 或会议接收信息。

---

### 3. 背景与动机：为什么需要这篇论文？

解释：

* 这个方向的背景；
* 当前常见 pipeline；
* 为什么已有方法不够；
* 论文试图解决的关键断点。

如果论文有 motivation figure，应插入在这里。

每张图都需要说明：

* 对应论文 Figure 编号；
* 图想表达什么；
* 它支持了哪个核心观点；
* 读者应该重点看哪里。

---

### 4. 核心问题：论文真正要解决的瓶颈

不要只复述 abstract，需要提炼：

* 输入是什么；
* 输出是什么；
* 当前 baseline 的失败模式是什么；
* 为什么这个失败模式不是简单加模型规模或数据就能解决；
* 论文的建模假设是什么。

建议用一个小结框总结：

* 问题本质；
* 技术难点；
* 作者的解法方向。

---

### 5. 方法总览：先给读者一张地图

在进入细节前，必须先解释完整 pipeline。

包括：

* 方法总览图；
* 输入、输出；
* 中间表示；
* 训练阶段；
* 推理阶段；
* 各模块之间的数据流；
* 方法相对 baseline 的关键差异。

如果论文有 framework figure，应插入在这里。

本节目标是让读者在看模块细节前先知道：

> 整个方法从哪里开始，经过哪些步骤，最后如何得到结果。

---

### 6. 核心方法精读

这是全文最重要的部分。

不要把“关键公式”“设计选择分析”“图表证据”单独拆成平级章节。它们应该穿插在本章对应模块的讲解中。

每个关键模块都按以下逻辑展开：

1. 设计动机：这个模块解决什么问题；
2. 技术机制：它具体怎么做；
3. 输入 / 输出：输入是什么，输出是什么，中间表示是什么；
4. 训练目标：相关 loss 或 objective 如何定义；
5. 推理过程：推理时如何使用；
6. 为什么有效：相比 baseline 解决了什么瓶颈；
7. 公式解释：如有关键公式，直接放在该模块内部解释；
8. 图表证据：如有对应 Figure / Table，直接插入该模块附近；
9. 实验证据：论文用什么实验或消融证明它有效；
10. 潜在局限：该模块可能失败的场景或工程风险。

公式要求：

* 使用 MathJax 渲染 LaTeX；
* 支持行内公式 `\( ... \)`；
* 支持独立公式块 `\[ ... \]`；
* 不要用代码块或纯文本模拟公式；
* 公式主展示区必须直接包含 MathJax 分隔符内的 LaTeX，例如 `<div class="math">\[ ... \]</div>`；不要只用 HTML 的 `<sub>/<sup>` 或普通文本模拟主公式；
* 如保留 LaTeX 源码 fallback，只能放在次要说明行，且不要再包 `\( ... \)`，避免源码说明行被渲染成主公式；
* 每个关键公式都标注对应论文 Equation 编号；
* 如果是为了理解而改写，必须标注“为便于理解的整理形式”。

每个关键公式都需要解释：

* 对应模块；
* 输入是什么；
* 输出是什么；
* 核心变量含义；
* 在训练或推理 pipeline 中的作用；
* 直觉解释；
* 工程实现时如何理解；
* 与 baseline 的差异；
* 对应实验或消融证据；
* 潜在局限。

MathJax 注意事项：

* 不要使用裸露的 `<` 或 `>`；
* 不要写 `z_{<i}`，应写 `z_{1:i-1}`；
* 不要写 `d<\delta`，应写 `d \lt \delta`；
* 检查是否存在公式空白、截断、溢出、括号不匹配、`\left` / `\right` 不匹配、重复 `\tag` 等问题。

如果论文包含多个模块，建议组织为：

#### 6.1 模块 A：名称

* 设计动机；
* 技术机制；
* 输入 / 输出；
* 关键公式与训练目标；
* 在 pipeline 中的位置；
* 与 baseline 的差异；
* 实验证据；
* 潜在局限。

#### 6.2 模块 B：名称

同上。

#### 6.3 模块 C：名称

同上。

#### 6.x 方法设计小结表

在方法章末尾加入一个简短的“设计选择小结表”，用于帮助读者回顾，不要替代正文解释。

表格至少包含：

| 设计选择 | 解决的问题 | 技术机制 | 实验证据 | 潜在局限 |
| -------- | ---------- | -------- | -------- | -------- |

表格内容要具体，避免泛泛而谈。

不要写：

> 提升性能。

应该写：

> 通过引入 X，使模型能够利用 Y 信息；在 Table N 的 ablation 中移除该模块后指标下降 Z，说明该设计对某类能力有贡献。

---

### 7. 实验设置：先教读者如何读实验

实验设置不能只列 dataset 和 metric，需要解释：

* benchmark 是什么；
* train / val / test split；
* 输入输出设置；
* baseline 分组；
* 主要 metric 含义；
* 哪些指标越高越好；
* 哪些指标越低越好；
* 哪些指标不能跨 benchmark 直接比较；
* 实验主要想验证哪些问题。

建议加一个“如何读这些指标”的小框：

* 核心指标是什么；
* 辅助指标是什么；
* 结果表中最应该关注哪几列；
* 不同 benchmark / stage / setting 之间是否可直接比较。

---

### 8. 主要实验结果

结果章节要按“结论”组织，而不是按表格编号机械堆叠。

推荐结构：

#### 结果 1：总体性能是否优于 baseline？

插入主结果表。

说明：

* 对应论文 Table 编号；
* 表格想说明什么；
* 本文方法相对 strongest baseline 的提升；
* 哪些指标提升最明显；
* 这个结果支持了哪个核心结论。

#### 结果 2：困难场景或泛化场景是否有效？

同上。

#### 结果 3：效率、速度或资源开销如何？

如果论文有相关结果，需要单独解释。

表格优先用 HTML 表格复刻；复杂大表可以用局部裁剪图片。

---

### 9. 消融分析：按“验证问题”组织

消融不要按 Table 3、Table 4、Table 5 顺序机械罗列，而要按每个 ablation 想验证的问题组织。

例如：

#### Q1：核心模块是否真的有用？

* 对应 Table / Figure；
* 对比哪些 variant；
* 去掉模块后发生什么；
* 说明该模块贡献了什么能力；
* 是否存在替代解释。

#### Q2：某个输入设计是否必要？

同上。

#### Q3：训练策略是否有效？

同上。

如果论文包含多个 model version、baseline 或 variant，需要解释它们各自的实验目的：

* 哪些 variant 验证数据贡献；
* 哪些 variant 验证架构贡献；
* 哪些 variant 验证训练目标贡献；
* 哪些 variant 验证输入设计贡献；
* 哪些 variant 验证推理机制贡献。

消融表应放在对应论证附近，不要集中堆成图表章节。

---

### 10. 可视化结果与失败案例

如果论文有 qualitative results、visualization、case study 或 failure cases，应单独解释。

本节重点不是堆图，而是回答：

* 可视化展示了什么能力；
* 它是否支持定量结果；
* 它揭示了哪些失败模式；
* 这些失败模式说明了什么局限；
* 对工程部署有什么启发。

每张图都需要说明：

* 对应论文 Figure 编号；
* 图中应该重点看哪里；
* 支持哪个核心结论；
* 暴露了什么问题或风险。

如果某张图更适合放在背景、方法或实验章节，应放到对应章节，不要为了集中展示而强行放在这里。

---

### 11. 优点、不足与隐含假设

这一节需要有分析，不要只写“效果好、结构简单”。

包括：

#### 优点

* 方法层面；
* 实验层面；
* 工程层面；
* 对研究方向的意义。

#### 不足

* 数据依赖；
* benchmark 局限；
* 模型假设；
* 计算开销；
* 推理延迟；
* 泛化风险；
* 失败案例；
* 作者没有充分讨论的问题。

#### 隐含假设

例如：

* 未来状态预测是否足够可靠；
* 训练分布和部署分布是否一致；
* offline metric 是否能代表 closed-loop 表现；
* 模块间误差是否会累积；
* teacher / annotator / simulator 是否足够可靠；
* benchmark 是否覆盖真实部署风险。

---

### 12. 对后续研究 / 工程实践的启发

面向研究者和工程落地分别总结。

研究启发：

* 可以进一步验证什么；
* 哪些设计可以迁移到其他任务；
* 哪些问题仍然开放；
* 后续工作可以如何改进。

工程启发：

* 如果要复现，最关键的模块是什么；
* 哪些模块可能带来算力或延迟压力；
* 哪些设计可能影响部署稳定性；
* 哪些指标需要额外监控；
* 哪些失败模式需要额外测试。

---

### 13. 总结

用清晰的几句话总结：

* 论文最核心贡献；
* 最有说服力的证据；
* 最大局限；
* 适合后续沿哪个方向继续研究。

---

### 14. 附录：PDF 解析与图表抽取自检

自检信息不要放进主线正文，也不要干扰阅读。可以放在页面底部附录或折叠区域。

包括：

* PDF 是否成功读取；
* 正文是否成功解析；
* 章节结构是否成功识别；
* Figure / Table / Equation 是否成功定位；
* 抽取了哪些 Figure / Table；
* 哪些表格被 HTML 复刻；
* 哪些复杂图表使用图片裁剪；
* 是否存在未能抽取的图表；
* 图表裁剪是否保留坐标轴、legend 和关键文字；
* 是否使用 MathJax；
* 是否检查公式渲染错误；
* 是否检索 Project / GitHub / OpenReview / Conference；
* 未找到的信息是否如实标注。

## 3. HTML 页面风格要求

1. 整体风格简洁、学术、美观；
2. 桌面端使用左侧固定导航栏；
3. 导航栏固定在页面最左侧；
4. 正文区域仍然保持居中；
5. 不要让正文整体偏右；
6. 不要让导航栏遮挡正文；
7. 点击导航项可以跳转章节；
8. 移动端可以把导航变成顶部横向导航或折叠导航；
9. 正文区域保持合适宽度和留白；
10. 图文混排，不要大段文字堆叠；
11. 章节层级清晰；
12. 重要结论使用 callout / card / summary box 强调；
13. 表格要可读，不要过窄或溢出；
14. 公式区域使用浅色背景、圆角边框、居中排版；
15. 不要把生成过程的自检信息放进主导航正文。

可读性硬性要求：

* 正文字号 16-18px，行高 1.65-1.85，正文最大宽度建议 880-980px；
* 不要使用过窄正文栏，不要让左侧导航挤压正文；
* 导航宽度建议 180-220px，正文和导航之间必须有明确留白；
* 中文正文优先使用系统无衬线或清晰宋黑搭配，不要使用导致中文粗细不均的装饰字体；
* 首屏必须先给结论、贡献和阅读路线，不要把自检、工具链、日志或长资源表放在首屏；
* 章节标题要短，段落要有小标题、callout、表格或列表分隔，避免连续大段文字；
* 图表说明必须写在图下方，先解释“怎么看”，再解释“说明什么”；
* 附录和自检默认折叠，不要默认展开占据主要阅读区域；
* 移动端和嵌入 iframe 时必须可读，不出现横向滚动正文。

HTML head 中加入 MathJax 配置和 CDN 脚本。如果 MathJax 加载失败，页面仍应显示可读的 LaTeX 源码。

---

## 4. 图表与表格处理要求

1. 不要嵌入整篇 PDF；
2. 不要使用 iframe / object / embed 加载 PDF；
3. 只抽取少量关键 Figure / Table；
4. 图表应插入对应论证位置，不要独立堆成“图表章节”；
5. 表格优先用 HTML 表格复刻；
6. 复杂大表可以用图片裁剪；
7. 每个图表必须有解释；
8. 每个图表必须说明对应论文编号；
9. 每个图表必须说明支持哪个核心结论；
10. 图片必须使用 base64 内嵌（`data:image/png;base64,...`），确保 HTML 离线可打开，不要使用外部 URL 或本地相对路径引用图片；
11. HTML 必须可离线打开，所有图片必须以 base64 内嵌（`<img src="data:image/...;base64,...">`），严禁引用外部图片 URL；
12. 不要把整篇 PDF 编码进 HTML；
13. 如果无法将图片以 base64 内嵌（例如裁剪失败），必须在自检清单中明确标注"图片未内嵌"，并说明原因。
14. 如果使用 SVG 概念图，必须内联 `<svg>` 或使用 `data:image/svg+xml;base64,...`，并在 caption 中写明“概念示意图，非论文原图”；
15. SVG 文字必须使用真实文本，不要把中文转成路径，不要出现编码乱码；
16. SVG 箭头和连线必须逐条检查方向，禁止线条穿过文字、穿过节点主体或连接到错误模块；
17. 复杂架构图优先使用论文原图裁剪；除非结构非常简单且已完全确认，否则不要手工重绘；
18. 图表区域必须设置 `max-width: 100%`、`height: auto`，小屏下可以横向滚动图表容器，但正文不能横向滚动；
19. 表格复刻必须保留指标方向、单位和最好结果标识；不得把不同 benchmark 的数值混在同一列比较。

## 8. 进度日志上报

在研究过程中，codex 应通过 `POST /api/research/{task_id}/log` 接口实时上报进度，让前端用户能看到研究进展。

请求体格式：

```json
{
  "text": "日志文本描述",
  "type": "info",
  "status": "researching"
}
```

字段说明：

- `text`（必填）：进度描述文本
- `type`（可选，默认 info）：日志类型，取值：
  - `system` — 系统消息（如"任务已创建"）
  - `search` — 搜索相关（如"正在搜索 arXiv..."、"找到相关论文"）
  - `reading` — 阅读分析（如"正在读取 PDF..."、"分析论文结构..."、"提取图表..."）
  - `generating` — 生成相关（如"正在生成精读 HTML..."、"渲染公式..."）
  - `info` — 一般信息
  - `error` — 错误信息
- `status`（可选）：同时更新任务状态，取值 `pending | researching | generating | completed | failed`

建议上报时机：

1. 开始搜索论文时 → `type: "search", status: "researching"`
2. 找到 PDF/论文页面时 → `type: "search"`
3. 开始读取 PDF 时 → `type: "reading"`
4. 分析论文结构/提取图表时 → `type: "reading"`
5. 开始生成 HTML 时 → `type: "generating", status: "generating"`
6. HTML 生成完毕，调用 complete 时 → 自然完成

完成后仍需调用 `POST /api/research/{task_id}/complete` 提交最终 HTML。
